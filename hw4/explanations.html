On Bootstrap

This site was built with the Bootstrap framework, which provides lots (and lots) of CSS and JavaScript fluff to build "modern" responsive webpages fairly easily. Compared to the WordPress version of the website, this site was actually far, far easier to build. Direct, total control of the markup goes a long, long way, as does the ability to override undesired CSS attribute settings as needed. (Even as it stands, I resorted to WordPress' "raw" HTML editor for that version of this site.) Some things (such as producing a general layout and style that I considered appropriate) were merely easier, but others (such as the slightly off-white background) were simply not feasible in WordPress, at least not without loads of effort, on the level of building a whole new WordPress theme. At least part of the trouble stems from the simple fact that WordPress is still really a blog platform; it's really, really not meant to do things like this site, even though that is done so frequently.

The only problem I can see really is that an absolutely non-technical user would be intimidated by working with raw HTML. As difficult as it is to get WSIWYG site-builders to behave (and as easy as it is to pick up enough basic HTML to cobble together a page like this), apparently many people would rather fight with the site-builder. I'm not sure I can say "and that's okay" with a straight face, but it's certainly The Way Things Are.

Most of Bootstrap's functionality is attached to individual HTML elements by specifying one or (many...) more of a number of pre-defined classes. This site's main page uses the following (not an exhaustive list):

  • container: The basis of the layout grid; all content goes in a container.
  • row: Placed inside containers to denote a section with more than one column.
  • col-*: A whole family of classes that denote a single column. The * represents one of the five "breakpoints", specifically the largest viewport size at which the layout will still render with columns side-by-side. col-md, for example, will render side-by-side only at sizes larger than "medium" (768 pixels wide).
  • display-4: One of a family of "display heading" styles, larger than the normal heading styles.
  • lead: Applied to the leading header paragraph, making it stand out more.
  • card, card-*: Cards are a form of content block with nicely pre-defined border and header image styles, among other things. The main page uses cards for the three "services" subsections.
  • m-*: A family of utility classes to add margins; the * is replaced with a small integer (the largest this site uses is, as I recall, m-4).
  • bg-*: A family of utility classes to set an element's background colors; the * is replaced with names like "primary", "secondary", "light", or "dark". This site uses only bg-primary and bg-light (and overrides the color of bg-light).
  • text-white: One of a family of utility classes to set an element's foreground/text color.
  • w-*: A family of utility classes to set an element's width, in increments of 25%.
  • border: Specifies that an element should have a border.
  • rounded: Specifies that an element should have rounded corners.
  • text-center: One of a family of styles to specify the text alignment in a block element.
  • img-fluid: The primary "responsive image" class, which ensures that an image is never larger than the screen, among other things; applied to the main header image, along with w-100, to simply specify that it fills the entire width of the screen.
